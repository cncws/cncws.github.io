RDB 持久化

更新于 2020-08-22 分类于 书籍 ， Redis 设计与实现 阅读次数： Valine：

Redis 是内存数据库，如果不想办法将储存在内存中的数据库状态保存到磁盘里面，那么一旦服务器进程退出，服务器中的数据库状态也会消失不见。

Redis 提供了 RDB 持久化功能，可以将内存中的数据库状态保存到磁盘里面，避免数据意外丢失。

RDB 持久化既可以手动执行，也可以根据服务器配置选项定期执行。

RDB 持久化功能所生成的 RDB 文件是一个经过压缩的二进制文件，通过该文件可以还原生成 RDB 文件时的数据库状态。

RDB 文件的创建与载入

有两个命令可以用于生成 RDB 文件，一个是 SAVE，另一个是 BGSAVE。

SAVE 命令会阻塞 Redis 服务器进程，直到 RDB 文件创建完毕为止。BGSAVE 会派生出一个子进程，然后由子进程负责创建 RDB 文件，服务器进程（父进程）继续处理命令请求。但是，在 BGSAVE 命令执行期间，服务器处理 SAVE、BGSAVE、BGREWRITEAOF 三个命令的方式会和平时有所不同：

  • SAVE：被拒绝。服务器禁止 SAVE 和 BGSAVE 同时执行是为了避免父进程和子进程同时执行两个 rdbSave 调用，防止产生竞争条件。
  • BGSAVE：被拒绝。因为同时执行两个 BGSAVE 也会产生竞争条件。

  • BGREWRITEAOF：如果 BGSAVE 正在执行，那么 BGREWRITEAOF 会被延迟到 BGSAVE 执行完毕之后执行；如果 BGREWRITEAOF 正在执行，那么 BGSAVE 会被拒绝。

    BGSAVE 和 BGREWRITEAOF 两个命令的实际工作都由子进程执行，所以这两个命令在操作方面并没有冲突的地方，不能同时执行它们只是一个性能方面的考虑——并发出两个子进程，并且这两个子进程都同时执行大量的磁盘写入操作，这怎么想都不会是一个好主意。

RDB 文件的载入是在服务器启动时自动执行的，所以 Redis 并没有专门用于载入 RDB 文件的命令。服务器在载入 RDB 文件期间，会一直处于阻塞状态。

另外，因为 AOF 文件的更新频率通常比 RDB 文件的更新频率高，所以如果服务器开启了 AOF 持久化功能，那么服务器会优先使用 AOF 文件来还原数据库状态。只有在 AOF 持久化功能处于关闭状态时，服务器才会使用 RDB 文件来还原数据库状态。

自动间隔性保存

Redis 允许用户通过设置服务器配置的 save 选项，让服务器每隔一段时间自动执行一次 BGSAVE 命令。用户可以设置多个保存条件，只要其中任意一个条件被满足，服务器就会执行 BGSAVE 命令。

服务器程序会根据 save 选项所设置的保存条件，设置服务器状态 redisServer 结构的 saveparams 属性。

RDB 文件结构

RDB 文件的开头保存着 "REDIS" 五个字符。

databases 包含零个或多个数据库，以及各个数据库中的键值对数据。如果服务器的数据库状态为空（所有数据库都是空的），那么这个部分也为空，长度为 0 字节。

EOF 常量标志着 RDB 文件正文内容的结束。

check_sum 是一个 8 字节长的无符号整数，是程序通过对前四个部分的内容计算得出的。服务器在载入 RDB 文件时，会将载入数据所计算出的校验和与记录的校验和进行对比，以此来检查 RDB 文件是否有出错或者损坏的情况出现。

databases 部分

key 总是一个字符串对象。TYPE 记录了 value 的类型，值可以是以下常量的其中一个：

  • REDIS_RDB_TYPE_STRING
  • REDIS_RDB_TYPE_LIST
  • REDIS_RDB_TYPE_LIST_ZIPLIST
  • REDIS_RDB_TYPE_SET
  • REDIS_RDB_TYPE_SET_INTSET
  • REDIS_RDB_TYPE_ZSET
  • REDIS_RDB_TYPE_ZSET_ZIPLIST
  • REDIS_RDB_TYPE_HASH
  • REDIS_RDB_TYPE_HASH_ZIPLIST

每个 TYPE 常量都代表了一种对象类型或者底层编码，程序根据 TYPE 的值来决定如何读入和解释 value 的数据。

value 的编码

字符串对象

如果 TYPE 的值为 REDIS_RDB_TYPE_STRING，那么 value 保存的就是一个字符串对象，字符串对象的编码（这里说的是字符串的编码，不是在 RDB 文件中的编码）可以是 REDIS_ENCODING_INT 或者 REDIS_ENCODING_RAW。

  1. 字符串的编码是 REDIS_ENCODING_INT，ENCODING 的值可以是 REDIS_RDB_ENC_INT8、REDIS_RDB_ENC_INT16 或者 REDIS_RDB_ENC_INT32 三个常量中的一个，分别代表 RDB 文件使用 8 位、16 位或者 32 位来保存整数值。

  2. 字符串的编码是 REDIS_ENCODING_RAW，根据字符串长度的不同，有压缩（前提是服务器打开了 RDB 文件压缩功能）和不压缩两种方法来保存：

    • 如果长度小于等于 20 字节，那么字符串会直接被原样保存。

    • 如果长度大于 20 字节，那么字符串会被压缩之后再保存。

      REDIS_RDB_ENC_LZF 常量标志着字符串已经被 LZF 算法压缩过了，读入程序在碰到这个常量时，会根据之后的三部分，对字符串进行解压缩。

列表对象

如果 TYPE 的值为 REDIS_RDB_TYPE_LIST，那么 value 保存的就是一个 REDIS_ENCODING_LINKEDLIST 编码的列表对象。item 的编码方式取决于它的类型。

集合对象

如果 TYPE 的值为 REDIS_RDB_TYPE_SET，那么 value 保存的就是一个 REDIS_ENCODING_HT 编码的集合对象。

哈希表对象

如果 TYPE 的值为 REDIS_RDB_TYPE_HASH，那么 value 保存的就是一个 REDIS_ENCODING_HT 编码的集合对象。

有序集合对象

如果 TYPE 的值为 REDIS_RDB_TYPE_ZSET，那么 value 保存的就是一个 REDIS_ENCODING_SKIPLIST 编码的有序集合对象。

INTSET 编码的集合

如果 TYPE 的值为 REDIS_RDB_TYPE_INTSET，那么 value 保存的就是一个整数集合对象，RDB 文件保存这种对象的方法是，先将整数集合转换为字符串对象，然后将这个字符串对象保存到 RDB 文件里面。

读入 RDB 文件时，碰到由整数集合对象转换成的字符串对象，先读入字符串对象，再根据 TYPE 值的指示转换成原来的整数集合对象。

ZIPLIST 编码的列表、哈希表或有序集合

TYPE 的值为 REDIS_RDB_TYPE_LIST_ZIPLIST、REDIS_RDB_TYPE_HASH_ZIPLIST 或者 REDIS_RDB_TYPE_ZSET_ZIPLIST。

RDB 保存这种对象的方法是（与 INTSET 类似）：

  1. 将压缩列表转换成一个字符串对象。
  2. 将转换所得的字符串对象保存到 RDB 文件。
