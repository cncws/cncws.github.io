生产者概览

Kafka 发送消息的主要步骤：

  1. 创建 ProducerRecord 对象，需要包含目标主题和消息内容，可以指定键或分区。发送对象时，生产者要先把键和值对象序列化成字节数组，这样才能在网络上传输。
  2. 数据被传给分区器。如果在对象里指定了分区，那么分区器就不会再做任何事情。如果没有指定分区，那么分区器就会根据键来选择一个分区。
  3. 接着，这条记录被添加到一个记录批次里，这个批次里的所有消息会被发送到相同的主题和分区上。有一个独立的线程负责把这些记录批次发送到相应的 broker 上。
  4. 服务器收到消息后，如果消息成功写入 Kafka，就返回一个 RecordMetaData 对象，它包含了主题和分区信息，以及记录在分区里的偏移量。如果写入失败，则会返回一个错误。生产者在收到错误之后会尝试重新发送消息，几次之后如果还是失败，就返回错误信息。

创建 Kafka 生产者

Kafka 生产者有 3 个必选的属性。

  • bootstrap.servers：broker 的地址清单，地址格式为 host:port。不需要包含所有的 broker 地址，生产者会自动发现。不过建议至少要提供两个 broker 地址，一旦其中一个宕机，生产者仍然能够连接到集群上。
  • key.serializer：必须被设置为一个实现了 org.apache.kafka.common.serialization.Serializer 接口的类，生产者会使用这个类把键对象序列化成字节数组。key.serializer 是必须设置的，就算你打算只发送值内容。
  • value.serializer：将值序列化。

发送消息主要有 3 种方式：

  • 发送并忘记：把消息发送给服务器，但并不关心它是否正常到达。大多数情况下，消息会正常到达（生产者会自动尝试重发）。
  • 同步发送：使用 send() 方法发送消息，它会返回一个 Future 对象，调用 get() 方法等待，就可以知道消息是否发送成功。
  • 异步发送：调用 send() 方法，并指定一个回调函数，服务器在返回响应时调用该函数。

同步发送

KafkaProducer 一般会发生两类错误。可重试错误，这类错误可通过重发消息来解决。比如对于连接错误，可以通过再次建立连接解决。KafkaProducer 可以被配置成自动重试，如果在多次重试后仍无法解决问题，应用程序会收到一个重试异常。另一类错误无法通过重试解决，比如“消息太大”异常。对于这类错误，KafkaProducer 不会进行重试，直接抛出异常。

异步发送消息

大多数时候，我们并不需要等待响应——尽管 Kafka 会把目标主题、分区信息和消息的偏移量发送回来，但对于发送端的应用程序来说不是必需的。不过在遇到消息发送失败时，我们需要抛出异常、记录错误日志，或者把消息写入“错误消息”文件以便日后分析。

为了在异步发送消息的同时能够对异常情况进行处理，生产者提供了回调支持。

生产者的配置

Producer Configs

  1. acks：必须要有多少个分区副本收到消息，生产者才会认为消息写入是成功的。
    • acks=0：生产者不会等待服务器的响应。无法保证服务器成功收到消息，同时 retries 配置会失效。
    • acks=1：默认配置。只要首领节点收到消息，生产者就会收到成功响应。收到错误响应时，生产者会重发消息。不过，如果一个未同步的节点成为新首领，消息还是会丢失。
    • acks=all：只有当所有参与复制的节点全部收到消息时，生产者才会收到一个来自服务器的成功响应。
  2. buffer.memory：生产者内存缓冲区的大小。当消息超出缓存时，send() 方法要么被阻塞（max.block.ms），要么抛出异常。
  3. compression.type：消息的压缩算法。可以设置为 none, gzip, snappy, lz4 或 zstd。默认为 none。
  4. retries：重发消息的次数。在 max.in.flight.requests.per.connection 设置不为 1 时，重发可能会导致消息的顺序改变。例如，先后发送了两个消息，而第一个消息失败了。
  5. batch.size：一个批次可以使用的内存大小（字节）。不过生产者并不一定都会等到批次被填满才发送。
  6. linger.ms：生产在发送批次之前等待更多消息加入批次的时间。生产者会在批次填满或 linger.ms 达到上限时把批次发送出去。
  7. client.id：可以是任意的字符串，服务器会用它来标识消息的来源。
  8. max.in.flight.requests.per.connection：生产者在收到服务器响应之前可以发送多少消息。把它设为 1 可以保证消息是按照发送的顺序写入服务器的。
  9. max.block.ms：调用 send() 方法或使用 partitionsFor() 方法获取元数据时生产者的阻塞时间。在阻塞时间达到 max.block.ms 时，生产者会抛出异常。
  10. max.request.size：控制生产者发送的请求大小。
  11. receive.buffer.bytes 和 send.buffer.bytes：TCP socket 接收和发送数据包的缓冲区大小。如果被设为 -1，就是用操作系统的默认值。

在 Kafka 里使用 Avro

Apache Avro

Avro 的数据文件里包含了整个 schema。如果在每条 Kafka 记录里都嵌入 schema，会让记录的大小成倍地增加。

使用 schema 注册表：把所有写入数据需要用到的 schema 保存在注册表里，然后在记录里引用 schema 的标识符。负责读取数据的应用程序使用标识符从注册表里拉取 schema 来反序列化记录。

分区

键有两个用途：作为消息的附加信息；决定消息该被写到主题的哪个分区。

如果键为 null，并且使用了默认的分区器，那么记录将被随机地发送到主题内各个可用的分区上。分区器使用轮询算法（Round Robin）将消息均衡地分布到各个分区上。

同一个键总是被映射到同一个分区上。只有在不改变主题分区数量的情况下，键与分区之间的映射才能保持不变。如果要使用键来映射分区，那么最好在创建主题的时候就把分区规划好，而且永远不要增加新分区。

可通过实现 org.apache.kafka.clients.producer.Partitioner 接口自定义分区器。
