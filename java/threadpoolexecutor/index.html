ThreadPoolExecutor

Pool Control State

ctl（Pool Control State，迷之缩写）是线程池中的一个变量 private final AtomicInteger ctl。如你所见，该变量是一个 AtomicInteger 类型的整数值。

ctl 打包了两个信息：工作线程数量 workerCount 和线程池运行状态 runState。一个整数如何打包两个信息呢？Java 使用整数的高 3 位保存 runState，低 29 位保存 workerCount。因此，理论上可以保存 8 种运行状态和 2^29 - 1 个工作线程。对现阶段来说，这个容量是足够的。同时文档中也提到了扩展：当容量不足时，可以使用 AtomicLong 代替 AtomicInteger。

在线程池内部经常会同时判断线程池状态和工作线程数量，若分别使用两个变量存储，则为了一致性就不得不使用锁。

runState

ctl 的初始化中使用了线程池的一个状态：RUNNING。线程池一共有 5 种工作状态（这也是为什么使用 3 位来保存 runState 的原因）：

状态描述
RUNING可接受新任务，以及处理工作队列中的任务。
SHUTDOWN不接受新任务，但是处理工作队列中的任务。
STOP不接受新任务，不处理工作队列中的任务，同时中断正在处理的任务。
TIDYING所有任务已结束，工作线程数量为 0 时，线程池状态将变为 TIDYING，同时触发 terminated() 方法。该方法的默认实现为空。
TERMINATEDterminated() 方法完成后，线程池状态将变为 TERMINATED。

阻塞队列的大小为 0 是 runState 从 SHUTDOWN 转换到 TIDYING 的条件之一。线程池使用 workQueue.isEmpty() 来判断是否为空。而不是 workQueue.poll() == null。这样做的目的是照顾某些特殊的队列，比如 DelayQueue，即使在不为空时，poll() 方法也可能返回 null。

拒绝策略

当线程池的工作者数量达到最大容量，且任务队列也满了时，就需要拒绝之后的任务。拒绝策略是一个接口：

用户可以实现该接口自己定义拒绝策略，或者使用 Java 内置的 4 种策略：

名称描述
AbortPolicy丢弃任务并抛出 RejectedExecutionException 异常。这是默认的拒绝策略。
DiscardPolicy丢弃任务但不抛出异常。一些无关紧要的业务可以使用此策略。
DiscardOldestPolicy丢弃队列中最老的任务（queue.poll() 返回的任务），然后重新尝试提交任务。
CallerRunsPolicy由调用线程（提交任务的线程）处理该任务。

Executors

Executors 工具类中提供了几种预实现的线程池，常用的有：

  • FixedThreadPool: 固定线程数量的线程池。
  • SingleThreadExecutor: 只有一个线程的“线程池”。
  • CachedThreadPool: 不限线程数量的线程池。
